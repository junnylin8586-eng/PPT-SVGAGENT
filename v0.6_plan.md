## 一、需求概述

三个迭代功能点：

| # | 功能 | 描述 |
|---|------|------|
| F1 | LLM API Key 用户配置 | 前端+后端，用户可在界面上配置 API Key 并验证连通性 |
| F2 | AI 人机对话生成 PPT 大纲 | 在主题描述输入界面增加对话输入框，用户与 AI 多轮对话生成大纲 |
| F3 | 模板配置 | 上传参考 PPT，AI 分析设计风格生成自定义模板，与现有模板系统融合 |

---

## 二、技术现状

### 后端（已有）
- `GET/PUT /api/settings` → Settings 模型（20+ 字段，含 per-model API 凭证）
- `POST /api/settings/check` → API Key 连接验证（已实现，provider 仅 minimax）
- `NewProjectModal.tsx` Step 1 → `analyzeTheme` SSE（单轮 AI 分析）

### 后端（缺失）
- 无 Settings 前端页面
- 无多轮对话大纲接口
- 无 PPT 解析和风格分析服务

### 前端（已有）
- `NewProjectModal.tsx`（Step 1 有简单 AI 分析 UI）
- `api/client.ts`（API 调用封装）

### 前端（缺失）
- `SettingsModal` 或 `SettingsPage`
- 多轮对话 UI
- 自定义模板上传 UI

### Phase 3：Feature 3 - 模板配置（上传 PPT 提取设计风格）

**目标**：用户上传成品 PPT → AI 分析风格 → 生成自定义模板 → 可用于新项目

#### 步骤 3.1：后端 PPT 解析服务
- [ ] 创建 `backend/services/ppt_parser.py`
  - 用 python-pptx 解析 PPT（字体、颜色、布局）
  - 截图每页缩略图（Playwright 或 pptx 导出）
  - 提取幻灯片母版样式

#### 步骤 3.2：后端设计风格 AI 分析
- [ ] 创建 `backend/services/style_analyzer.py`
  - 调用 MiniMax API 分析截图
  - 输出设计元数据（primary_color, font_title, font_body, layout_type, mood）

#### 步骤 3.3：自定义模板存储
- [ ] 创建 `backend/template_custom/{uuid}/`
  - `preview.png`（封面缩略图）
  - `meta.json`（设计元数据）
  - `pages/`（每页截图）
  - `template_info.json`（名称、分类、描述）
- [ ] 数据库：`layout_templates` 表扩展（is_custom, source_pptx, design_meta）

#### 步骤 3.4：后端接口
- [ ] `GET /api/templates/custom` → 获取自定义模板列表
- [ ] `POST /api/templates/upload` → 上传 PPT，分析风格
- [ ] `DELETE /api/templates/{id}` → 删除自定义模板
- [ ] `GET /api/templates/{id}/preview` → 预览图

#### 步骤 3.5：前端自定义模板 UI
- [ ] 在模板选择页增加"自定义模板" Tab
- [ ] 上传 PPT UI（拖拽上传、进度条）
- [ ] 模板预览和删除功能

## 四、技术难点与注意事项

### F1 - Settings
- [ ] API Key 安全：前端不存储明文，显示掩码
- [ ] `.env` vs 数据库：优先级策略需明确（数据库覆盖 .env 或反之）

### F2 - AI 对话
- [ ] MiniMax-M2.7 是否支持 function calling/tool use，需验证
- [ ] 多轮对话 context 长度控制
- [ ] 对话历史超过 10 轮需做截断

### F3 - PPT 解析
- [ ] python-pptx 解析字体/颜色依赖 PPT 内部元数据，部分模板样式定义在母版中可能读取不到
- [ ] 降级方案：AI 分析截图兜底
- [ ] PPT 文件大小限制（建议 ≤ 50MB）
